# Pitfalls Research

**Domain:** 암호화폐 스캘핑 봇 — 틱 데이터 수집 + 백테스트 인프라
**Researched:** 2026-05-19
**Confidence:** HIGH (이미 이 시스템에서 발생한 버그 기반 + 퀀트 백테스팅 도메인 공통 실패 패턴)

---

## Critical Pitfalls

### Pitfall 1: WebSocket 무음 단절 → 틱 갭 미탐지

**What goes wrong:**
빗썸 WS가 끊겨도 봇은 "연결 정상"으로 인식하고 계속 로깅한다. 수집기는 끊긴 구간의 틱을 빈 배열이 아니라 **아예 기록하지 않는다** — 즉 DB에는 갭이 보이지 않고, 연속된 것처럼 보이는 가짜 연속 시계열이 생긴다. 이 데이터를 백테스트에 쓰면 갭 구간에서 "신호 없음"이 아니라 "마지막 정상 틱이 지속된 것"처럼 해석된다. 실제 가격이 급등/급락한 구간이 평탄하게 뻥튀기된다.

이 프로젝트의 기존 코드(`alt_monitor.py` PriceTracker)에 자동 재연결 로직이 없고, 로그에 "458개 코인 추적" 반복 출력이 WS 사망 이후에도 나온다는 것이 이미 CONCERNS.md에서 확인된 버그다.

**Why it happens:**
`websocket-client` 라이브러리는 연결 단절을 자동 재시도하지 않는다. heartbeat/ping-pong 없이 연결 객체만 살아있으면 "connected"로 본다. 수집기를 새로 만들 때 같은 구조를 복사하면 동일 버그가 재현된다.

**How to avoid:**
- 틱 수집기에 타임스탬프 기반 갭 감시 스레드를 독립 추가: 마지막 틱 수신 후 N초(예: 5초) 내 신규 틱 없으면 "WS dead" 플래그 → 자동 재연결 + 경고
- 갭 구간을 DB에 명시적 NULL 행 또는 별도 `tick_gaps` 테이블로 기록 (갭 시작 timestamp, 종료 timestamp, 이유)
- 백테스트 엔진은 갭 구간을 포함한 이벤트를 "데이터 없음"으로 처리, 해당 구간 신호 생성 금지

**Warning signs:**
- 1분 이상 같은 가격이 이어지는 코인 (WS 정지 시 마지막 값 고정)
- 로그에 "458개 코인 추적" 주기 깨지지 않는데 틱 DB insert 수가 급감
- 특정 시간대(새벽 서버 점검)에 데이터가 완전히 비는 패턴

**Phase to address:**
Phase 1 (틱 수집기 구축) — 수집기를 처음 만들 때 갭 감지 + 재연결을 함께 구현. 나중에 붙이면 기존 데이터의 갭 오염 범위를 알 수 없다.

### Pitfall 2: 스냅샷 타임스탬프 불일치 → 가짜 동시성

**What goes wrong:**
빗썸 REST API 응답의 타임스탬프(서버 시간)와 수집기 로컬 시계(Windows `datetime.now()`)가 수십~수백 ms 다르다. WS 틱은 수신 시각이 아니라 거래소 이벤트 시각이어야 한다. 이를 혼용하면 백테스트에서 이벤트 순서가 뒤집힌다: 실제 가격 변동이 먼저 일어났는데, 수집 지연 때문에 신호 계산보다 늦게 기록된다. 결과적으로 백테스트는 "신호 발생 → 0.2초 후 진입"인데 실제는 진입 시점보다 가격이 이미 0.5초 이상 앞서 움직인 것이다.

**Why it happens:**
Python `datetime.now()`는 수집기 머신 로컬 시계다. Windows 시스템 시계는 수십 ms 드리프트가 일상적이다. 빗썸 WS 이벤트 필드에 거래소 timestamp가 있으나, 이를 파싱하지 않고 수신 시각으로 덮어쓰면 정밀 시뮬레이션이 불가능하다.

**How to avoid:**
- 틱 DB 스키마에 `exchange_ts`(거래소 발생 시각), `recv_ts`(수집기 수신 시각) 두 컬럼 분리 저장
- 백테스트에서 이벤트 순서 정렬 기준은 `exchange_ts` 사용
- 수집기 시작 시 NTP 동기화 상태 점검 (Windows `w32tm /query /status` 또는 `ntplib`)

**Warning signs:**
- 같은 코인에서 `recv_ts < exchange_ts` 행이 나타남 (시계 역전)
- 틱 간격이 이론값(WS이면 실거래마다)보다 훨씬 들쭉날쭉
- 백테스트 진입 가격이 신호 시점 가격보다 낮게 나오는 케이스가 통계적으로 많음

**Phase to address:**
Phase 1 (틱 수집기 구축) — 스키마 설계 시 두 타임스탬프 분리가 원칙. 나중에 ALTER TABLE로 추가하면 기존 데이터 소급 불가.

### Pitfall 3: Lookahead Bias — 미래 정보가 과거 신호 판정에 침투

**What goes wrong:**
백테스트 코드가 `peak_price`, `max_drawdown` 같은 이벤트 후 집계값을 신호 필터 조건에 쓰는 순간 lookahead bias가 발생한다. 이미 이 프로젝트에서 확인된 버그(`peak_at_sec` 같은 미래 정보를 진입 판정에 사용)다. 현실에서는 진입 시점에 미래 고점을 알 수 없으므로, 백테스트 승률은 실거래보다 훨씬 높게 나온다.

더 미묘한 형태: "이 코인이 5분 후 +10% 갔다"는 pump_log를 필터로 쓰는 경우 — 데이터 선별 자체가 lookahead bias다.

**Why it happens:**
pump_log/틱 데이터에는 이벤트 경로 전체(진입 이전 + 이후)가 저장된다. 백테스트 코드를 작성할 때 "이 코인 이벤트에서 max가 뭐였더라"를 편의상 미리 계산해 조건에 쓰면 즉시 lookahead가 된다.

**How to avoid:**
- 백테스트 엔진에 시간 커서(time cursor) 강제 — 커서 이전 데이터만 노출하는 `DataSlice` 추상화 구현. 커서 이후 칼럼에 접근하면 AssertionError
- `peak_price`, `recovery_at_sec` 같은 사후 집계 컬럼을 별도 테이블(`tick_outcomes`)에 분리, 백테스트 런타임에는 이 테이블을 JOIN 금지
- 코드 리뷰 체크리스트에 "이 값은 진입 시점에 알 수 있었나?" 필수 항목 추가

**Warning signs:**
- 백테스트 승률 > 60% (실거래 25%와 큰 차이 — 이 갭은 거의 항상 lookahead)
- 신호 필터에 `max_`, `peak_`, `final_`, `outcome_` prefix 변수가 들어 있음
- 백테스트 결과가 파라미터와 무관하게 안정적으로 좋음 (데이터 자체가 편향된 것)

**Phase to address:**
Phase 2 (백테스트 엔진 설계) — `DataSlice` 또는 `TimeBarrier` 클래스를 엔진의 핵심 추상화로 처음부터 설계. 이것을 건너뛰고 "일단 돌아가는" 백테스트를 만들면 이후 수정이 매우 어렵다.

### Pitfall 4: 청산 모델링 낙관 편향 — 진입 후 경로 없이 EXIT 시뮬레이션

**What goes wrong:**
스냅샷(1분/2분/3분/5분) 기반 데이터로는 진입 후 가격이 얼마나 내려갔다가 올라왔는지 경로를 알 수 없다. 백테스트가 "5분 스냅샷 +3% → 익절 성공"으로 처리하지만, 실제로는 2분에 -5%로 손절됐을 수 있다. 이 프로젝트의 기존 분석에서 명시된 "청산 모델링 낙관 편향"이 바로 이것이다.

틱 데이터를 수집해도, 틱 간격이 1~3초인데 손절/익절 판정을 1분 봉 기준으로 하면 동일한 bias가 재현된다.

**How to avoid:**
- 초 단위 틱으로 청산 시뮬레이션: 매 틱마다 stop/tp 조건 검사 (`for tick in ticks_after_entry: if tick.price <= stop_price: exit()`)
- 슬리피지 모델 필수: 청산 주문 제출 → 체결까지 최소 1~2초 딜레이 + 체결가는 청산 틱 가격이 아니라 다음 틱 가격 사용
- 부분 체결 모델링: 시장가 매도 시 호가 깊이에 따라 다른 가격으로 체결 (빗썸 소형 코인은 호가 깊이 매우 얕음)

**Warning signs:**
- 1분 봉 기반 백테스트 PnL이 틱 기반 백테스트 PnL보다 유의미하게 높음 (정상은 틱이 더 낮아야 함)
- 이론 손절가 -3%인데 실거래 평균 손실이 -6%인 경우 (갭 슬리피지 + 경로 bias)

**Phase to address:**
Phase 2 (백테스트 엔진) — 틱 단위 청산 검사를 기본 동작으로, 봉 단위는 "빠른 근사값 전용"으로 명시.

### Pitfall 5: 그리드 서치 과적합 — 표본 내 최적 파라미터의 허상

**What goes wrong:**
수집한 틱 데이터(예: 3주, 코인 N개)로 PULLBACK_TARGET_PCT, HOLD_MIN_SEC 등 6개 파라미터를 그리드 서치하면 표본 내 수익률이 극대화된 조합이 나온다. 그러나 그 조합이 실제 미래에서도 동작한다는 보장이 없다. 이 프로젝트에서 이미 지적된 함정 — 350건에 다파라미터 그리드 서치 → 과적합.

데이터가 적을수록 (2일·36코인) 파라미터 1개당 필요 독립 표본이 부족하다. 자유도(파라미터 수) > 표본 수 / 10 이면 과적합 확률이 거의 확실하다.

**How to avoid:**
- 데이터를 처음부터 시간 순으로 3분할: train(60%) / validation(20%) / test(20%)
- 파라미터 서치는 train만 사용, validation으로 조기 종료, test는 최종 1회만 사용
- Walk-forward 검증: train 윈도우를 1주 단위로 슬라이딩, 각 윈도우 외부에서 성과 확인
- 파라미터 수를 최소화: 검증할 수 있는 최대 파라미터 수 = 총 거래 수 / 30 (경험칙)

**Warning signs:**
- 그리드 서치 최적 조합의 train 승률 >> validation 승률 (10%p 이상 차이)
- 최적 파라미터가 그리드 범위의 극단값(최소 또는 최대)에 위치
- 서로 다른 랜덤 시드로 서치할 때마다 최적 파라미터가 크게 달라짐

**Phase to address:**
Phase 3 (전략 검증) — 데이터 분할 구조를 수집 직후에 확정. 검증 전에 test 데이터를 들여다보면 오염된다.

### Pitfall 6: 슬리피지 완전 누락 → EV 2~3%p 낙관 편향

**What goes wrong:**
수수료(0.25% × 2 = 0.5%)만 반영하고 슬리피지를 0으로 두면, 신규 상장 코인이나 급등 코인의 실제 거래 비용이 2~3배 과소 추정된다. 빗썸 소형 알트코인은 호가 스프레드가 0.5~2%이고, 시장가 매수 시 1호가가 아니라 2~3호가까지 먹히는 경우가 흔하다. 이미 이 프로젝트 분석에서 "슬리피지 누락 → EV 2~3%p 낙관"이 명시됐다.

**How to avoid:**
- 백테스트 거래 비용 모델: 수수료(0.5% 왕복) + 슬리피지(1.0% 보수적 기본값) + 시장충격(거래금액 × 0.1%) 합산
- 호가 데이터 수집: 틱 수집기에 `GET /public/orderbook/{coin}` 스냅샷을 10초마다 함께 저장, 시장가 체결 시뮬레이션을 실제 호가 깊이로 계산
- 슬리피지 민감도 분석: 슬리피지 0% / 0.5% / 1% / 2% 시나리오별 EV를 리포트에 포함, "어느 슬리피지까지 양수 EV인가" 기준선 제시

**Warning signs:**
- 백테스트 평균 진입가가 신호 발생 시 시세와 동일 (슬리피지 없음의 증거)
- EV가 양수인데 실거래에서 계속 손실 (이 프로젝트가 이미 경험한 상황)
- 거래 직후 즉시 -1~2%인 포지션 다수 (체결 슬리피지)

**Phase to address:**
Phase 2 (백테스트 엔진) — 거래비용 모델을 엔진 초기화 파라미터로, 기본값을 보수적으로 설정. "슬리피지 = 0" 옵션은 연구 목적 이외 금지.

### Pitfall 7: 표본 편향 — 펌핑 코인만 수집해 생존자 편향 발생

**What goes wrong:**
현재 수집기가 "펌핑 감지 → pump_log 저장" 구조인 경우, DB에는 감지된 펌핑 이벤트만 들어있다. 아무 일도 없었던 코인, 조용히 하락한 코인은 기록되지 않는다. 이 상태에서 전략을 검증하면 "이미 오른 코인 중 얼마나 계속 올랐나"를 보는 것이 되어, 실제 "신호를 받은 코인 전체 중 얼마나 올랐나"보다 훨씬 낙관적으로 나온다.

**Why it happens:**
이벤트 드리븐 수집은 관심 이벤트만 저장하는 것이 자연스럽다. 그러나 전략 검증은 신호 발생 시 기대값(양수/음수 포함 전체)을 봐야 한다.

**How to avoid:**
- 기록 전용 모드(record-only)에서 모든 코인의 틱을 저장 — 신호를 받든 안 받든
- 신호 발생 기록(`signal_log`)과 가격 경로(`tick_log`)를 분리해 독립적으로 저장. 신호 없이 가격 경로만 있는 행이 "노이즈" 표본이 됨
- 백테스트 쿼리: "신호 발생 시점에서 N초 후 가격" — 신호 필터를 통과한 것뿐 아니라 통과 직전에 기각된 것도 포함해 threshold 민감도 검증

**Warning signs:**
- DB에서 "signal이 없는 ticker 데이터"가 없음 (이벤트만 수집된 증거)
- 백테스트 표본 중 손실 이벤트 비율이 실거래 손실 비율보다 현저히 낮음

**Phase to address:**
Phase 1 (틱 수집기) — 저장 범위를 처음부터 "전체 코인, 전체 시간" 또는 "필터 전 신호 로그 포함"으로 설계.

### Pitfall 8: 페이퍼 트레이딩의 거짓 안심 — 성공이 실거래 보장하지 않음

**What goes wrong:**
페이퍼 트레이딩(모의거래)에서 성공한 전략이 실거래에서 즉시 손실 나는 가장 흔한 이유는 **시장 충격 부재**다. 페이퍼는 "원하는 가격에 언제나 체결"을 가정한다. 실거래에서 시장가 매수 주문이 들어오면 호가 잔량을 소모하고, 다음 호가로 넘어가며 체결가가 낮아진다. 빗썸 소형 알트에서 30,000원 매수가 1호가 잔량을 다 먹으면 2호가로 넘어가는 일이 흔하다.

추가로, API 레이턴시: 페이퍼는 신호 발생과 동시에 "체결"이지만 실거래는 신호→HTTP POST→체결 확인까지 150~500ms. 이 시간 동안 가격이 변한다.

**How to avoid:**
- 페이퍼 모드에도 레이턴시 시뮬레이션: 신호 발생 후 `time.sleep(uniform(0.15, 0.5))` 후 "체결"
- 체결가는 신호 시점 가격이 아니라 그 이후 수신된 다음 WS 틱 가격 사용
- 페이퍼에서 KRW 30,000 이상 주문 시 1~2% 슬리피지를 강제 적용
- 페이퍼 → 실거래 전환 기준: 페이퍼 성과가 (예상 슬리피지 2배 패널티 적용 후)에도 양수 EV

**Warning signs:**
- 페이퍼 승률 >> 백테스트 승률 (레이턴시/슬리피지 없어서)
- 페이퍼 체결가가 항상 신호 발생 시 마지막 가격과 완전 일치

**Phase to address:**
Phase 4 (페이퍼 트레이딩) — 페이퍼 구현 시 레이턴시 + 슬리피지 모델을 백테스트와 동일하게 공유.

### Pitfall 9: 통계적 유의성 착각 — 표본 부족에서 결론 도출

**What goes wrong:**
36~100건 거래 결과로 "이 전략은 EV 양수"를 선언하면 표본 오차가 너무 크다. 예를 들어 30건에서 승률 40%라면 95% 신뢰구간이 약 ±18%p다 — 즉 실제 승률이 22~58%라는 뜻이다. 이 프로젝트에서 "초기 4건 행운거래(+57K)가 음의 EV를 가렸다"는 사례가 정확히 이 함정이다.

**How to avoid:**
- 검증 결론 도출 전 최소 표본 기준 사전 확정: 예) "승률 추정 오차 ±10%p 이내면 결론 도출 → 최소 96건 필요"
- 신뢰구간을 항상 리포트에 포함: 승률 40% (95% CI: 31%~50%, n=70)
- 누적 PnL 차트 대신 rolling 승률 차트를 메인 지표로 — 초기 행운거래의 영향을 시각적으로 포착
- 결론 선언 기준을 코드로: `if n < MIN_SAMPLE or ci_width > MAX_CI_WIDTH: raise InsufficientDataError`

**Warning signs:**
- "10건 해봤는데 승률 70%, 전략 확정"
- PnL이 초기 몇 건에 의존 (처음 3건 제거하면 결론이 반전)
- 파라미터 변경 후 "5건 보니까 괜찮아 보여요" 즉흥 결론

**Phase to address:**
Phase 3 (전략 검증) — 검증 사이클 진입 전에 "최소 표본 수, CI 기준, 결론 선언 조건"을 문서로 확정.

---

### Pitfall 10: 다중 검정 문제 — 여러 전략 중 "좋아 보이는 것" 선택

**What goes wrong:**
20개 파라미터 조합을 테스트해서 가장 좋은 1개를 고르면, 그 조합이 우연히 좋을 확률이 5% 유의수준에서 훨씬 높아진다. 5% 확률 이벤트를 20번 시도하면 64%에서 "좋아 보이는" 조합이 등장한다. 이를 "진짜 발견"으로 착각하면 그것이 Bonferroni 함정(다중 검정 문제)이다.

**How to avoid:**
- 파라미터 조합 수(N)에 맞춰 유의수준 조정: α_adjusted = 0.05 / N (Bonferroni 보정)
- 검증 기준: "최고 조합 1개"가 아니라 "상위 K개 조합 모두 out-of-sample에서 양수 EV"
- 가설은 사전에 등록: 서치 전에 "이 조합이 좋을 것"을 써놓고, 서치 후 결과를 확인 (HARKing — Hypothesizing After Results are Known 방지)

**Warning signs:**
- 서치 공간이 클수록 최적값이 "너무 깔끔하게" 나옴
- Out-of-sample 성과가 in-sample 대비 급격히 하락 (10%p 이상)

**Phase to address:**
Phase 3 (전략 검증) — 검증 체계 설계 시 다중 검정 보정 방법론을 사전에 문서화.

---

### Pitfall 11: 틱 DB 스키마 초기 설계 실수 → 마이그레이션 비용

**What goes wrong:**
수집 시작 후 2주가 지나서 "호가 데이터도 필요했다", "exchange_ts가 없다", "coin_id 대신 coin_name을 저장했는데 대소문자 불일치"를 발견하면 기존 데이터 전체가 불완전한 상태가 된다. SQLite ALTER TABLE은 컬럼 추가만 되고, 기존 행의 새 컬럼은 NULL이므로 과거 데이터는 사용 불가.

**Why it happens:**
"일단 수집하고 나중에 스키마 다듬자"는 유혹. 틱 데이터는 2주 기다려야 하므로 스키마 실수의 비용이 일반 DB보다 훨씬 크다.

**How to avoid:**
- Phase 1 착수 전 스키마 리뷰 체크리스트 필수 통과:
  - `exchange_ts` / `recv_ts` 분리 ✓
  - 코인 식별자 통일 (대문자, BTC/KRW 분리 ✓)
  - 결측값 허용 여부 명시 (price NOT NULL, volume NOT NULL) ✓
  - 호가 스냅샷 저장 여부 결정 ✓
  - 인덱스 사전 설계 (coin, exchange_ts) ✓
- 수집 시작 전 더미 데이터 1시간으로 백테스트 엔진 연결 테스트 — 스키마 결함 조기 발견

**Warning signs:**
- 백테스트 엔진 코드 작성 중 "이 컬럼이 없네"를 처음 발견
- 코인 이름이 DB에 BTC, btc, BTC_KRW 혼재

**Phase to address:**
Phase 1 시작 전 설계 검토 — 수집 코드 1줄 쓰기 전에 스키마를 확정하고 사용자 동의.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 스냅샷(1분봉) 기반 백테스트 먼저 구현 | 빠른 구현, 틱 수집 전에 엔진 구조 검증 가능 | 청산 경로 bias, 슬리피지 모델 불가 — 틱 엔진으로 전환 시 전면 재작성 | 엔진 구조 POC 한정, 절대 결론 도출에 사용 금지 |
| 단일 SQLite 파일에 틱 + 신호 + 거래 혼용 | 쿼리 편의 | 2주 후 파일 수백 MB, 틱 쓰기와 신호 읽기가 락 경합 | 수집 초기 1~2일 테스트까지만 허용 |
| 수집기에 재연결 없이 "일단 돌려보기" | 당장 수집 시작 | 첫 WS 단절 시 갭이 발생하고 언제인지 모름 | Never — 갭은 기록할 수 없는 데이터 오염 |
| 백테스트에서 슬리피지 = 0 | EV 계산 간단 | EV 2~3%p 낙관, 실거래 전환 시 즉시 손실 | 상대 비교(전략 A vs B)에만, 절대 EV 판단 금지 |
| pump_log의 기존 4개 스냅샷 재사용 | 기존 코드 변경 없음 | 진입 후 경로 불완전, 손절 시뮬레이션 불가 | Never — 이번 마일스톤의 핵심 목표가 이 교체 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| 빗썸 WS pubwss | ping_interval 없이 연결 → 서버 idle timeout 후 무음 단절 | `websocket.WebSocketApp`에 `ping_interval=20, ping_timeout=10` 설정 + on_error/on_close 핸들러에서 재연결 트리거 |
| 빗썸 REST `/public/ticker/ALL` | 전체 코인 응답을 "실시간"으로 착각 — 실제는 1~3초 캐시 | 고빈도 수집에는 WS 사용, REST는 초기화/검증 전용으로 한정 |
| SQLite WAL 모드 | 기본 모드에서 WS 쓰기 스레드 + 메인루프 읽기 경합 → SQLITE_BUSY | `PRAGMA journal_mode=WAL` 설정, 연결별 `timeout=5` |
| Windows 시스템 시계 | `datetime.now()` drift가 수십~수백 ms — 고빈도 틱 타임스탬프 부정확 | `datetime.now(timezone.utc)` + NTP 동기화 상태 확인 후 수집 시작 |
| 빗썸 JWT 인증 | `nbf` 클레임 없이 iat만 쓰면 서버 시간 1초 오차로 401 | iat를 `int(time.time())` 로 정확히 계산, 서버 시간과 주기적 동기화 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 458코인 × 틱마다 DB INSERT (동기) | WS 수신 → INSERT 지연 증가, 큐 백로그 | 수신 스레드는 in-memory 큐에만 push, 별도 writer 스레드가 배치 INSERT (`executemany`) | 초당 10틱 이상 (빗썸 활성 시간대) |
| 백테스트에서 전체 틱 DataFrame 메모리 로드 | 2주 데이터 로드 시 2~8GB RAM 사용, OOM | 코인별/날짜별 청크 로딩, generator 기반 시뮬레이션 | 틱 데이터 1주일 초과 시 |
| 백테스트 결과 계산 중 pandas apply() 남용 | 전략 1회 백테스트에 10분 이상 소요 | 벡터화 연산(numpy), 또는 Cython/numba 핵심 루프 최적화 | 파라미터 그리드 서치 10×10 이상 |
| 틱 DB에 인덱스 없음 | 코인별 쿼리에 수 초 소요 (백테스트 체감 속도 붕괴) | `CREATE INDEX idx_ticks_coin_ts ON ticks(coin, exchange_ts)` 스키마 생성 시 포함 | 틱 행 10만 건 초과 시 |

## "Looks Done But Isn't" Checklist

- [ ] **WS 재연결**: 연결 코드가 있어도 — "30분 WS 무음 후 자동 복구되는가" 직접 테스트 (케이블 뽑기 시뮬레이션)
- [ ] **갭 감지**: 수집기가 "갭을 기록하는가" vs "갭을 조용히 건너뛰는가" 확인 — `tick_gaps` 테이블 존재 여부
- [ ] **슬리피지 반영**: 백테스트 리포트에 "slippage model: 1.0%" 명시 여부 — 없으면 0으로 처리된 것
- [ ] **Lookahead 차단**: 백테스트 엔진이 커서 이후 컬럼에 접근하면 에러를 던지는가 — 단순 "미래 변수 안 썼어요"는 검증 안 됨
- [ ] **Out-of-sample 보호**: test 데이터셋을 검증 완료 전에 단 한 번도 쿼리하지 않았는가 — 파일 접근 로그 확인
- [ ] **표본 수 기준**: 결론 선언 전 `n >= MIN_SAMPLE` 조건을 코드가 강제하는가
- [ ] **실거래 차단**: 기록 전용 모드에서 `market_buy()` 가 절대 호출되지 않는가 — mock 주입이 아니라 코드 경로 자체 차단 여부

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| WS 갭 미탐지로 데이터 오염 발견 | HIGH | 오염 구간 특정 (WS 재연결 로그 vs 틱 timestamp 갭 비교) → 해당 구간 틱 삭제 → 재수집 (최소 1주 소요) |
| Lookahead bias 발견 후 전략 결론 번복 | HIGH | 백테스트 엔진 TimeBarrier 재구현 → 전체 파라미터 서치 재실행 → 이전 결론 무효 선언 |
| 스키마 오류 (exchange_ts 누락) | HIGH | 새 스키마로 테이블 재생성 → 수집 재시작 → 2~3주 재대기 |
| 슬리피지 0으로 "EV 양수" 결론 후 실거래 손실 | MEDIUM | 슬리피지 1% 재적용 후 EV 재계산 → EV 음수면 전략 폐기 → 실거래 즉시 중단 |
| 과적합 최적 파라미터가 실거래에서 실패 | MEDIUM | Test 데이터 out-of-sample 재확인 → 유효한 파라미터 범위 좁힘 또는 전략 폐기 |
| 표본 부족에서 도출한 결론 번복 | LOW | 표본 추가 수집 후 신뢰구간 재계산 → 결론 유보 또는 갱신 |
