# Feature Research

**Domain:** 암호화폐 스캘핑 봇 — 틱 데이터 수집 + 백테스트/검증 인프라
**Researched:** 2026-05-19
**Confidence:** HIGH (기존 코드베이스 직접 분석 + 도메인 지식 기반)

---

## 맥락: 무엇을 만드는가

기존 시스템(WebSocket 시세, SQLite, REST API)은 그대로 두고, 그 위에 "검증 레이어"를 올린다.
목표는 딱 하나: **전략의 EV(기댓값)가 양수인지 음수인지 데이터로 증명한다.**

현재 pump_log는 10초마다 가격을 읽지만 4개(1/2/3/5분) 스냅샷만 저장하고 나머지 26개를 버린다.
이것이 정밀 백테스트가 불가능한 근본 원인이다. 틱 수집이 모든 것의 선행 조건이다.

## Feature Landscape

### Table Stakes — 없으면 검증 시스템이 무의미

| Feature | 왜 필수인가 | 복잡도 | 구현 메모 |
|---------|------------|--------|----------|
| **틱 데이터 수집** (초단위 가격 경로 저장) | 이게 없으면 나머지 모두 불가능. 현재 pump_tracker는 10초 루프로 이미 가격을 읽고 있음 — 버리지 말고 저장만 하면 됨 | LOW | 기존 pump_tracker `time.sleep(10)` 루프 안에서 tick 배열에 append → 5분 후 JSON/별도 테이블에 일괄 저장 |
| **기록 전용 모드 (dry-run flag)** | 실거래 위험 0으로 2~3주 데이터 축적. 수집 기간 중 실수로 주문 나가면 안 됨 | LOW | `config.yaml`에 `dry_run: true` 플래그. 주문 함수 진입부에서 차단 + 로그 출력 |
| **백테스트 엔진 코어** | 틱 데이터를 시간순으로 재생하며 전략 로직을 오프라인 실행 | MEDIUM | 틱 배열을 iterator로 feed → 진입 조건 평가 → 가상 포지션 관리 (진입가·익절가·손절가) → 청산 시 결과 기록 |
| **수수료 + 슬리피지 모델링** | 없으면 백테스트 결과가 실거래와 달라 의미 없음. 빗썸 0.25% 왕복 + 신규상장 슬리피지 최소 0.5~2% | LOW | 상수로 주입: `FEE_RATE=0.0025`, `SLIPPAGE=0.01` (보수적). 진입가에 슬리피지 더하고, 청산가에서 수수료 차감 |
| **핵심 지표 출력** | 결과가 숫자로 안 나오면 판단 불가. EV·승률·MDD·거래수 4개는 최소 | LOW | 백테스트 결과 dict에 포함: `win_rate`, `ev_per_trade`, `mdd`, `total_trades`, `total_pnl` |
| **Out-of-sample 분할** | Train으로 파라미터 고르고 Test로 검증 안 하면 오버피팅. 개인 프로젝트라도 이건 생략하면 안 됨 | LOW | 날짜 기준으로 단순 슬라이스. 예: 처음 2주 train / 마지막 1주 test. 코드 1줄 |

### Differentiators — 있으면 유용하지만 없어도 검증은 돌아간다

| Feature | 가치 제안 | 복잡도 | 구현 메모 |
|---------|----------|--------|----------|
| **파라미터 그리드 서치** | 손으로 파라미터를 하나씩 바꾸는 수고 없이 최적 구간을 탐색. 핵심 파라미터(진입 딜레이, 손절%, 익절%)에 한정 | MEDIUM | `itertools.product`로 파라미터 조합 생성 → 각 조합에 백테스트 실행 → EV 기준 정렬. 조합 수 200개 이하로 제한해야 실행 시간 현실적 |
| **페이퍼 트레이딩 모드** | 실시간 시세로 가상 거래를 시뮬레이션. 백테스트 통과 후 실거래 전 최종 검문소 역할 | MEDIUM | 기존 봇 코드에서 실제 주문 호출 제거 → 가상 포지션 dict 관리. dry_run과 다른 점: 실시간 시세로 실행, 결과를 별도 파일에 축적 |
| **코인별 통계 분리** | 전체 승률보다 코인별로 어떤 게 EV 양수인지가 핵심 인사이트 | LOW | 백테스트 결과를 coin 키로 groupby → 코인별 승률·EV 출력 |
| **진입 시간대별 분석** | 새벽 시간대 진입이 낮 시간대보다 나쁜지 데이터로 확인. STRATEGY.md에서 미결로 남긴 사항 | LOW | signal_log.hour_kst 컬럼 이미 존재. 백테스트 결과에 hour 컬럼 추가하면 됨 |

### Anti-Features — 개인 학습 프로젝트에서 일부러 배제

| Feature | 왜 요청받는가 | 왜 배제하는가 | 대신 할 것 |
|---------|-------------|-------------|-----------|
| **Walk-forward / 롤링 윈도우 최적화** | "더 정교한 백테스트" | 2~3주치 데이터로 롤링을 돌리면 각 윈도우당 샘플이 수십 건에 불과 → 통계적 의미 없음. 구현 복잡도만 높아짐 | 단순 train/test 2분할로 충분 |
| **Monte Carlo 시뮬레이션** | "드로우다운 분포 보고 싶다" | 57건 샘플에서 재샘플링하면 신뢰구간이 너무 넓어 행동 불가능한 숫자가 나옴. 오히려 잘못된 확신 줄 수 있음 | MDD 하나만 계산. 충분 |
| **멀티 심볼 동시 포지션 백테스트** | "여러 코인을 동시에 트레이딩하면?" | 현재 봇이 단일 포지션 설계임. 멀티 심볼 로직은 기존 코드를 뒤집어야 함. 이번 마일스톤 범위 밖 | 단일 심볼 백테스트로 전략 EV 검증에 집중 |
| **머신러닝 신호 모델** | "ML로 진입 예측하면 더 잘 되지 않을까?" | 데이터가 수십~수백 건. ML 학습에 필요한 수천 건 훨씬 못 미침. 오버피팅 100% | 규칙 기반 파라미터 최적화로 해결 가능한 범위에서 작업 |
| **웹 대시보드 / 실시간 모니터링 UI** | "시각화하면 좋겠다" | Flask/Dash 셋업 + 유지보수 비용. 이 마일스톤의 가치는 UI가 아니라 EV 판정 | CLI 출력 + CSV 덤프로 충분. 필요하면 엑셀/Google Sheets에서 열면 됨 |
| **실시간 파라미터 핫 리로드** | "봇 재시작 없이 파라미터 바꾸고 싶다" | 검증 체계에서는 파라미터를 즉흥적으로 바꾸는 행위 자체를 금지하고 있음 (STRATEGY.md, PROJECT.md 모두 명시) | 파라미터 변경 → 백테스트 → 결과 확인 → 반영 사이클 준수 |
| **거래소 API 기반 히스토리 데이터 수집** | "빗썸 OHLCV 다운로드해서 과거 데이터로 백테스트" | 빗썸 OHLCV는 캔들 단위(1분~). 이 봇의 진입·청산 판단은 초 단위 가격 경로에 의존. OHLCV로는 슬리피지·진입 타이밍을 재현 불가 — 잘못된 낙관적 결과 나옴 | 직접 수집한 초단위 틱 데이터만 사용 |

## Feature Dependencies

```
[기록 전용 모드 (dry_run)]
    └──선행조건──> [틱 데이터 수집]
                       └──선행조건──> [백테스트 엔진 코어]
                                          ├──선행조건──> [수수료+슬리피지 모델링]
                                          ├──선행조건──> [핵심 지표 출력]
                                          └──선행조건──> [Out-of-sample 분할]

[백테스트 엔진 코어] ──enables──> [파라미터 그리드 서치]
[백테스트 엔진 코어] ──enables──> [코인별 통계 분리]
[백테스트 엔진 코어] ──enables──> [진입 시간대별 분석]

[Out-of-sample 분할 + 그리드 서치 통과] ──gate for──> [페이퍼 트레이딩 모드]
[페이퍼 트레이딩 모드] ──gate for──> [실거래 재개 (다음 마일스톤)]
```

### Dependency Notes

- **틱 데이터 수집이 dry_run보다 선행해야 한다:** dry_run 없이도 틱 수집은 가능하지만, 틱 수집 없는 dry_run은 데이터를 안 모으면서 실거래만 막는 반쪽짜리 상태가 됨. 두 개를 한 번에 올려야 함.
- **수수료 + 슬리피지 모델링은 백테스트 엔진과 분리:** 상수 파라미터로 주입하면 나중에 숫자를 바꾸기 쉬움. 엔진 안에 하드코딩하지 말 것.
- **그리드 서치는 out-of-sample 분할 전에 train 셋에만 실행:** 그리드 서치가 test 셋을 보면 데이터 누출(leakage). 순서가 중요.
- **페이퍼 트레이딩은 백테스트 통과 후에만 의미:** 백테스트에서 EV 음수로 나왔는데 페이퍼 트레이딩 돌리는 것은 시간 낭비. 게이트 역할.

## MVP Definition

### Launch With (v1) — 틱 수집 + 기본 백테스트

이 네 가지가 있어야 "데이터 수집 → 검증"의 최소 사이클이 돌아간다.

- [ ] **기록 전용 모드 (dry_run flag)** — 실거래 위험 없이 2~3주 운영 가능해야 하므로
- [ ] **틱 데이터 수집** (초단위 가격 경로, DB 또는 JSON) — 이것이 없으면 나머지 전부 불가
- [ ] **백테스트 엔진 코어** + 수수료/슬리피지 모델링 — 틱 데이터가 쌓이는 동안 병행 제작 가능
- [ ] **핵심 지표 출력** (승률·EV·MDD·거래수) + **Out-of-sample 분할** — 결과 없으면 판단 불가

### Add After Validation (v1.x) — 인사이트 확장

기본 백테스트로 EV 방향성(양수/음수)이 확인된 뒤 추가.

- [ ] **파라미터 그리드 서치** — EV가 양수 구간이 존재할 때 최적 파라미터 찾기 위해
- [ ] **코인별 / 시간대별 통계 분리** — 전략 세분화 가능 여부 확인 시

### Future Consideration (v2+) — 실거래 재개 전 검문소

별도 마일스톤에서 판단.

- [ ] **페이퍼 트레이딩 모드** — 백테스트 통과 전략을 실시간 실거래 없이 최종 확인
- [ ] **실거래 재개** — PROJECT.md Out of Scope에 명시됨. 검증 결론 후 별도 마일스톤

## 핵심 지표 상세 스펙 (백테스트 출력 최소 요구사항)

백테스트 엔진이 반드시 계산해야 하는 4개:

| 지표 | 정의 | 왜 이것인가 |
|------|------|------------|
| **win_rate** | 수익 거래 수 / 전체 거래 수 | 현재 실거래 승률 ~25%. 기준점. |
| **ev_per_trade** | 총 PnL(수수료 포함) / 전체 거래 수 (원 단위) | 양수/음수가 GO/NO-GO 판정의 핵심 |
| **mdd** | 최대 연속 손실 기간의 누적 손실 (%) | 실거래 재개 시 리스크 한도 설정에 사용 |
| **total_trades** | 시뮬레이션 기간 내 발생한 총 거래 수 | 샘플 수가 너무 적으면 결과 자체를 신뢰하지 말 것 (최소 30건 이상) |

추가 권장 (구현 비용 LOW):
- `avg_hold_sec` — 보유 시간 분포 (HOLD_MIN_SEC 파라미터 적정성 검증)
- `exit_reason_dist` — 손절/익절/타임아웃 비율 (어디서 돈을 잃는지 파악)
- `max_consecutive_losses` — 연속 손실 최대 횟수 (심리적 버팀 가능 여부)
